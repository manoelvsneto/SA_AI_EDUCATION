| A | The integration of Artificial Intelligence (AI) into modern software architectures is perceived as the key to adding intelligence, autonomy, and optimization to distributed systems. This union not only increases the systems’ ability to handle complexity and scale, but also redefines the development cycle and the user experience.<br>AI is introduced into the studied architectures as follows:<br>Microservices: AI comes to life in dedicated Machine Learning (ML) services, which encapsulate predictive or generative models (such as recommendation, fraud detection, or classification). This modular approach allows the AI model to be developed, deployed, and scaled independently, adhering to the microservices principle.<br>Event-Driven Architecture (EDA): AI functions as a central event consumer or producer. An AI service consumes a business event (“order placed”) and, after real-time inference, publishes a new decision event (“high fraud risk”), triggering subsequent actions by other components. This gives the system agility and instant reaction capability.<br>Serverless: This architecture is ideal for AI inference workloads, especially those with demand peaks. Serverless functions (Function as a Service - FaaS) host model inference logic, scaling automatically and charging only for usage, becoming a cost-effective solution for executing AI models.<br>BFF (Backend For Frontend): AI acts to personalize the user experience. The BFF can use the output of AI services to adapt the layout, content, or functionality of a specific interface (web, mobile), or it can directly incorporate Generative AI to create conversational interfaces and interactive chatbots. | The use of architectural communication models and approaches (such as arc42, C4 Model, and UML) is fundamental to make tangible and align the complex decisions made in solutions involving the triad AI, Cloud, and Distributed Architectures (Microservices, Serverless, etc.). They help overcome the difficulty of communicating systems that are inherently abstract and dynamic, ensuring that the boundaries between the AI model, Cloud infrastructure, and application code are clear. 🎨 How Models Help Communication C4 Model: Optimizing the Hierarchical View The C4 Model (Context, Containers, Components, Code) is the most effective approach for communication at different audience levels: C4 Level Target Audience Contribution in AI/Cloud Solutions 1. Context Stakeholders, Business Defines the scope of the AI system (e.g., “Recommendation System”) and the boundaries with external systems, including critical Cloud services (e.g., Payment Gateway, Data Platform). 2. Containers Architects, Managers Shows the deployable units in the Cloud: Serverless functions that execute model inference, feature store microservices, cloud-managed databases, and event queues (Event-Driven). 3. Components Developers Details the internal structure of a container, such as an ML service, specifying the components that perform data preprocessing, model inference (the AI algorithm itself), and result post-processing. 4. Code Developers Optionally focuses on the implementation of classes and methods, being able to detail the AI model call interface. The C4 Model enables hierarchical zoom, ensuring that Business stakeholders understand the AI value proposition (Level 1) and Developers understand Cloud deployment (Level 2) and the internal design of AI services (Level 3). arc42: Complete Documentation Structure arc42 provides a complete documentation skeleton, ensuring that all architectural decisions are comprehensively recorded: Decisions and Justifications: Helps record the justification for using a specific AI model (e.g., why use a specific LLM or which ML algorithm), or the choice of a cloud service (e.g., using a Managed ML Service instead of virtual machines). Technical Requirements: Documents the performance and cost requirements of the AI model (e.g., maximum latency for inference, cost per prediction) and Cloud constraints (e.g., deployment region, instance type required for GPUs). Quality View: It is crucial for recording MLOps (Machine Learning Operations) aspects, such as monitoring model drift, security (access to training data), and scalability of AI components in cloud infrastructure. UML: Detailing Interaction and Logic Although the C4 Model is preferred for its simplicity, UML (Unified Modeling Language) is still useful for detailing specific aspects of complex systems: Sequence Diagrams: Essential to show the flow of synchronous calls between microservices and the AI service, or the sequence of events in an Event-Driven architecture, helping identify latency bottlenecks. Class/Component Diagrams: Can be used at the lowest level (C4 - Code/Component) to show the internal structure of an AI model and its software interfaces. In summary, these approaches create a standard visual language to communicate the static structure of the Cloud (Resources and Containers) and the dynamic behavior of the AI component (Data Flow and Inference), transforming complex decisions into an artifact understandable by all stakeholders. | Key Learning Factors<br>Modularity as an Enabler of AI (Microservices, Serverless, EDA):<br>The most significant learning was seeing how distributed architectures allow AI models to become decoupled and scalable components.<br>Instead of a monolith, AI is encapsulated in a dedicated Microservice or Serverless Function that only performs inference. This simplifies the model life cycle and allows it to be scaled and updated independently from the rest of the application.<br>Event-Driven Architecture (EDA) stood out as essential for real-time AI, allowing the system to react instantly to business data to make decisions (e.g., fraud).<br>Architectural Modeling as an Alignment Tool (C4 Model):<br>The C4 Model proved to be the most effective tool for communicating the complexity of Cloud and AI solutions.
